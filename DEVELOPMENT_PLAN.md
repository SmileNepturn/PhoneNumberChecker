# iPhone OCR 전화번호 DB 앱 개발 계획

## 1. 목표

아이폰에서 앱스토어 등록 없이 실행 가능한 프로그램을 만든다. 사용자는 업체명, 전화번호, 주소가 들어 있는 엑셀 출력물 또는 화면을 사진으로 찍고, 앱은 사진 속 텍스트를 OCR로 추출한 뒤 전화번호 중심의 DB를 만든다.

핵심 동작은 다음과 같다.

- 사진 촬영 또는 이미지 선택
- OCR로 업체명, 전화번호, 주소 추출
- 추출 후 원본 이미지는 저장하지 않고 폐기
- 전화번호를 기준으로 기존 DB와 중복 비교
- DB에 없는 전화번호만 사용자에게 하나씩 표시
- 사용자가 `부재중`, `거절`, `승락` 중 하나를 선택하고 `다음`을 누르면 DB에 저장
- 다음날 새 사진을 찍으면 `거절`, `승락` 전화번호는 다시 보여주지 않고 `부재중` 전화번호는 다시 보여줌

## 2. 배포 방식

앱스토어 등록 없이 아이폰에서 운영하려면 기본 방식은 PWA로 한다.

- Safari에서 접속 가능한 웹앱으로 개발
- 아이폰 Safari의 `홈 화면에 추가` 기능으로 앱처럼 실행
- 카메라 촬영은 `<input type="file" accept="image/*" capture="environment">`를 사용
- 초기 버전은 로컬 저장소 기반으로 동작
- 이후 여러 기기 동기화가 필요하면 서버 DB를 추가

## 3. 핵심 사용자 흐름

1. 사용자가 홈 화면 앱을 실행한다.
2. `사진찍기` 버튼을 누른다.
3. 아이폰 카메라로 업체 리스트 사진을 촬영한다.
4. 앱이 OCR을 실행해 사진 속 텍스트를 추출한다.
5. 추출된 텍스트에서 업체명, 전화번호, 주소 후보를 구조화한다.
6. 전화번호를 정규화한다.
7. 기존 DB에 같은 전화번호가 있는지 확인한다.
8. DB에 없는 전화번호만 처리 대기 목록에 넣는다.
9. 사용자에게 전화번호 1개와 업체명, 주소 후보를 보여준다.
10. 사용자가 `부재중`, `거절`, `승락` 중 하나를 선택한다.
11. `다음` 버튼을 누르면 해당 결과가 DB에 저장되고 다음 번호가 표시된다.
12. 대기 목록이 끝나면 처리 결과 요약을 보여준다.

## 4. 데이터 모델

### contacts

전화번호 기준으로 처리 이력을 저장한다.

| 필드 | 설명 |
| --- | --- |
| id | 내부 고유 ID |
| company_name | 업체명 |
| phone_number | 원본 전화번호 |
| normalized_phone | 숫자만 남긴 중복 비교용 전화번호 |
| address | 주소 |
| status | `missed`, `rejected`, `accepted` |
| source_date | 사진 처리 날짜 |
| created_at | 최초 저장 시각 |
| updated_at | 마지막 수정 시각 |
| ocr_raw_text | OCR 원문 일부 또는 전체 |

### import_sessions

사진 1장 또는 1회 OCR 실행 단위의 기록을 저장한다.

| 필드 | 설명 |
| --- | --- |
| id | 세션 ID |
| imported_at | 사진 처리 시각 |
| image_name | 이미지 파일명 또는 임시명 |
| total_detected | OCR에서 찾은 전화번호 수 |
| duplicate_count | 기존 DB와 중복된 수 |
| pending_count | 새로 처리해야 할 수 |
| saved_count | 사용자가 처리 완료한 수 |

## 5. 전화번호 정규화 규칙

중복 판단은 표시용 전화번호가 아니라 정규화된 전화번호로 한다.

- 숫자 외 문자 제거
- `02`, `031`, `010`, `0507` 등 국내 번호 패턴 지원
- 하이픈, 공백, 괄호가 달라도 같은 번호로 인식
- 예: `02-123-4567`, `02 123 4567`, `(02)123-4567`은 모두 `021234567`
- 한국 전화번호만 대상으로 처리

## 6. OCR 및 텍스트 구조화 전략

초기 버전은 구현 난이도와 배포 편의성을 고려해 브라우저 기반 OCR을 우선 검토한다.

### 1차 구현 후보

- Tesseract.js
- 아이폰 Safari에서 이미지 OCR 실행
- OCR worker, wasm core, 한국어/영어 학습 데이터를 앱에 포함
- OCR 실행 전 이미지 확대, 회색조, 대비 강화, 이진화 전처리 적용
- 사용자가 미리보기에서 시계방향/반시계방향 90도 회전으로 정렬한 뒤 OCR 실행
- 사용자가 미리보기에서 OCR 대상 영역을 드래그 선택하면 해당 영역만 OCR 처리
- 장점: 설치 후 서버 의존 없이 OCR 실행 가능
- 단점: 성능이 느릴 수 있고 표 형태 인식 정확도가 제한적

### 2차 개선 후보

- 서버 API 기반 OCR
- Google Vision API, Naver CLOVA OCR, AWS Textract 등 검토
- 장점: 표 형태와 한글 인식 정확도 개선
- 단점: 네트워크, 비용, 개인정보 처리 고려 필요

### 구조화 방식

1. OCR 원문을 줄 단위로 분리한다.
2. 각 줄에서 전화번호 패턴을 먼저 찾는다.
3. 전화번호가 있는 줄 주변 텍스트를 업체명, 주소 후보로 묶는다.
4. 주소 키워드와 지역명 패턴을 이용해 주소 후보를 보정한다.
5. 사용자가 보는 화면에서는 OCR 결과를 수정할 수 있게 한다.

## 7. 화면 구성

### 홈 화면

- `사진찍기` 버튼
- 최근 처리 날짜
- 전체 DB 등록 수
- 오늘 처리한 수

### OCR 확인 화면

- 촬영 이미지 미리보기
- OCR 진행 상태
- 추출된 전화번호 개수
- 기존 DB 중복 개수
- 새로 처리할 전화번호 개수
- `검토 시작` 버튼

### 전화번호 처리 화면

- 업체명
- 대표 전화번호
- 주소
- OCR 신뢰도 또는 원문 보기
- `부재중` 버튼
- `거절` 버튼
- `승락` 버튼
- `다음` 버튼
- 필요 시 업체명, 전화번호, 주소 직접 수정 입력

### 이력 화면

- 날짜별 처리 목록
- 업체명 검색
- 전화번호 검색
- 상태별 필터: 전체, 부재중, 거절, 승락

## 8. 저장소 선택

### 초기 버전

- IndexedDB 사용
- 아이폰 Safari/PWA에서 로컬 DB처럼 사용 가능
- 별도 서버 없이 빠르게 개발 가능

### 확장 버전

- 서버 API + SQLite 또는 PostgreSQL
- 여러 아이폰 또는 PC에서 같은 DB를 공유해야 할 때 도입
- 백업과 엑셀 내보내기 기능 추가 가능

## 9. 개발 단계

### 1단계: 프로토타입

- PWA 프로젝트 생성
- 아이폰 카메라 촬영 입력 구현
- 촬영 이미지 미리보기
- Tesseract.js OCR 연결
- OCR 원문 화면 표시

### 2단계: 전화번호 추출

- 한국 전화번호 정규식 구현
- 전화번호 정규화 함수 구현
- OCR 원문에서 전화번호 후보 추출
- 중복 제거 로직 구현

### 3단계: DB 저장

- IndexedDB 스키마 생성
- contacts 저장/조회 구현
- normalized_phone 기준 중복 매칭 구현
- 처리 상태 저장 구현

### 4단계: 사용자 검토 플로우

- 새 전화번호만 큐로 생성
- 1개씩 표시하는 화면 구현
- `부재중`, `거절`, `승락`, `다음` 처리 구현
- 업체명, 전화번호, 주소 수정 기능 구현

### 5단계: 이력 및 검색

- 날짜별 처리 이력 화면 구현
- 업체명/전화번호 검색 구현
- 상태별 필터 구현
- 처리 통계 표시

### 6단계: 아이폰 실사용 검증

- Safari에서 카메라 촬영 테스트
- 홈 화면 추가 후 실행 테스트
- 오프라인 상태 동작 테스트
- OCR 속도와 정확도 확인
- 실제 엑셀 출력물 사진으로 반복 테스트

## 10. 주요 리스크와 대응

| 리스크 | 대응 |
| --- | --- |
| OCR이 표 형태를 정확히 읽지 못함 | 전화번호 우선 추출 후 주변 줄을 업체명/주소 후보로 묶음 |
| OCR 결과에서 전화번호가 추출되지 않음 | 사용자에게 재촬영 안내를 표시하고 검토 진입을 막음 |
| 아이폰에서 OCR 속도가 느림 | 이미지 리사이즈 후 OCR 실행, 필요 시 서버 OCR로 전환 |
| 같은 번호가 다른 형식으로 들어옴 | normalized_phone으로 중복 판단 |
| Safari 로컬 데이터가 삭제될 수 있음 | 백업/내보내기 기능 추가 |
| 업체명과 주소 매칭이 틀림 | 사용자가 저장 전 수정할 수 있게 함 |

## 11. 우선순위

MVP에서 반드시 필요한 기능은 다음으로 제한한다.

1. 아이폰에서 사진 촬영
2. OCR 원문 추출
3. 전화번호 추출 및 정규화
4. 기존 DB 중복 제외
5. 새 전화번호 1개씩 표시
6. `부재중`, `거절`, `승락` 저장
7. 날짜, 업체명, 전화번호, 주소 저장

업체명과 주소 자동 매칭은 처음부터 완벽하게 만들기보다, 전화번호 중심으로 먼저 안정화하고 사용자 수정 기능으로 보완한다.
